Schema Builder

Features

Schema Builder is a full-stack tool for turning unstructured documents (PDF, DOCX, PPTX) into structured records. The features below describe what ships today — the index links to a longer write-up further down.

Detailed descriptions

Visual definition editor

Definitions describe what to pull out of a document — a list of fields, each with examples, options, optional regex, and rendering hints. Originally these had to be uploaded as raw JSON; the editor is a Carbon-based modal form with live validation against the same Pydantic schema the backend enforces. It preserves extra keys it doesn't know about, so hand-written extensions survive a round-trip.

Starter templates library

Bundled starter JSONs cover common shapes — invoice, receipt, business card, purchase order, bank statement — and are served read-only via GET /api/templates. In the editor's create flow, a Start from template dropdown hydrates a fresh draft so users don't have to start from a blank document.

Version history with diff & restore

Every overwrite, PATCH, or DELETE snapshots the previous content to a per-definition versions directory with a timestamp-ms filename. The editor's History modal lists snapshots, renders a side-by-side recursive-sorted JSON diff between any version and the current state, and offers a one-click restore that PATCHes the older content back in.

Click-to-teach from the page

The rendered page overlays every text entry Docling found. Clicking one opens a small picker; choosing a field appends the selected text to that field's examples list and re-runs the extraction automatically. The loop between "this is wrong" and "fixed for next time" closes inside the viewer — no JSON editing required.

Per-field regex / pattern slot

Alongside examples and options, a field can declare a pattern. The matcher scores regex hits at 92 and uses capture group 1 (falling back to group 0) as the extracted value, so an IBAN pattern returns just the IBAN even when surrounded by other text. Pydantic refuses uncompilable regexes at upload time and the editor surfaces compile errors live as you type.

Per-field normalizers

Raw extracted text is rarely the value you want to store. A normalizer slot on each field transforms it — supported kinds include number, currency, date (optionally with a format), percent, boolean, trim, lowercase, and uppercase. Both rule-based and LLM-fallback matches pass through the normalizer; the panel shows the parsed value next to the raw text so the transformation is visible.

Field dependencies (visible_if / required_if)

Fields can declare visibility or required-ness conditional on other fields. The grammar supports {field, equals|in|present|absent} plus {all: [...]} / {any: [...]} combinators and a bare true for "always required". Suppressed fields are wiped post-match with reason hidden_by_dependency; required-but-missing fields surface a red required badge in the panel rather than silently passing.

Per-field confidence thresholds

Every field can override the default 0.5 confidence cutoff with a min_confidence between 0 and 1, exposed in the editor as a percent input. Sub-threshold candidates surface as a review hint rather than being dropped, so borderline matches stay visible for human triage instead of disappearing.

Docling-powered text & layout extraction

Extraction is built on Docling, which provides layout-aware text with bounding boxes. The accelerator (CUDA / MPS / CPU) is detected at startup and logged once so the answer to "is it actually using my GPU?" appears in the log without waiting for the first request. Thread count, OCR mode, and device can all be pinned via environment variables.

Auto-OCR routing for image-only PDFs

Two Docling pipelines are kept warm: a fast no-OCR converter and an OCR converter. For each document the backend samples extractable text with pypdfium2 and routes digital PDFs through the fast path and image-only PDFs through OCR automatically. The decision is surfaced on /metrics and can be forced with DOCLING_DO_OCR when the heuristic guesses wrong.

Optional LLM fallback per field

Each field carries a use_llm_fallback flag. When the rule-based matcher returns nothing for a flagged field, the backend calls Claude via the Anthropic SDK with the field description and the page text, using structured outputs and prompt caching. The SDK is lazy-imported so it isn't a hard dependency; the model and a global enable switch are env-controlled (SCHEMABUILDER_LLM_MODEL, SCHEMABUILDER_LLM_ENABLED). LLM-derived values are tagged LLM in the fields panel.

Multi-page tables with header detection

Array (table) fields can set multi_page and supply a header_pattern regex. Rows matching the header pattern are filtered out, and when multi-page is on, rows whose tokens are a subset of the sub-field-name tokens — i.e. the column header repeated on later pages — are auto-skipped. The result includes pages_spanned and is_multi_page so the UI can badge "pages 1–3" on the table.

Bounding-box overlay on rendered pages

Every match draws a persistent ghost overlay on the page PNG. Hovering a row in the fields panel highlights its overlay, and hovering an overlay highlights its row — the link runs in both directions. Active matches show a label with the field name and score so match_reason and match_score become tangible rather than abstract numbers in a table.

Batch extraction with progress

POST /api/extract/batch enqueues a job that runs sequentially behind the existing concurrency semaphore. GET /api/extract/batch/{id} returns progress for polling; DELETE cancels after the current document finishes. The sidebar exposes a Run all button that opens a modal with a progress bar, cancel control, and a JSON download of the aggregated results.

Persisted extraction cache (SQLite)

The full extraction response is cached on a key combining the document signature and a matcher-relevant definition hash. Edits to target_tables don't invalidate (they don't affect matching); deleting a document invalidates by signature. The cache survives restarts, is bounded by an LRU on created_at, and a ?refresh=true query bypasses it on demand.

Document upload & delete

POST /api/documents accepts uploads with filename sanitization, collision suffixing, and a separate 50 MB body cap independent of the regular request limit. DELETE /api/documents/{id} purges the render and text caches so a re-upload doesn't serve stale pages. The sidebar adds an Upload button and a per-row trash icon.

Target-table export (CSV / JSON)

Definitions can describe one or more target_tables — named outputs built from extracted fields via simple transforms (rename, join, format). The export endpoint executes those transforms and streams the result as CSV or JSON. The transform engine sits outside the matcher, so editing a target table doesn't invalidate the extraction cache.

Schema codegen (JSON Schema / SQL / TypeScript)

The same definition that drives extraction can be handed to a downstream consumer without a manual re-write. The editor's Export schema menu downloads the current definition as a JSON Schema document, a PostgreSQL or BigQuery CREATE TABLE block, or a TypeScript interface. Scalar types are inferred from the field's normalizer (number, currency, and percent become numeric; date becomes a date; boolean becomes boolean); available_options becomes an enum or string-literal union; type: array fields fan out into one child table (SQL) or a nested object array (JSON Schema / TS) so the output drops cleanly into a warehouse or a typed pipeline.

Dark mode & keyboard navigation

A theme toggle in the header flips between Carbon's g10 and g90 themes, persisted to localStorage with the OS prefers-color-scheme as the first-load default. Keyboard shortcuts cycle documents (j / ↓, k / ↑) and pages (← / →), all suppressed when focus is in a form control so they don't fight with typing.

Offline overlay when the backend is down

When the backend is unreachable, a Carbon overlay pauses the UI with a single steady reconnect indicator (intentionally one indicator, not flickering retry chrome). Reconnect is automatic and the overlay dismisses itself once health checks succeed again.

Health, readiness & metrics endpoints

/health is a true liveness probe — 200 whenever the process is up — and reports directory existence, warm-up status, and in-flight count. /ready is a separate readiness probe that returns 503 Retry-After: 5 until the no-OCR pipeline has loaded. /metrics reports cache hits/misses, OCR routing decisions, completed extractions, and concurrency rejections.

Concurrency limits & load shedding

/extract is bounded by a process-global semaphore (SCHEMABUILDER_MAX_CONCURRENT_EXTRACTS, default 4). Excess requests get 503 Retry-After: 5 rather than queuing behind the serialized Docling pipeline, so clients and any upstream load balancer can shed load instead of stacking up. The body size limit is enforced via Content-Length so oversized payloads never stream into memory.

OpenAPI snapshot drift check

openapi-snapshot.json is committed to the repo. A pytest case and a dedicated CI step diff the live FastAPI schema against the snapshot; if anything has shifted, the build fails with a clear run python export_openapi.py SNAPSHOT instruction. Frontend and backend can't drift on the API surface without someone noticing.

Per-request tracing IDs

Every response carries an X-Request-ID header — echoed from the inbound header if present, generated otherwise. Log lines emitted while handling a request are prefixed with the same id, so a single failed call can be traced end-to-end through the server log without grepping by timestamp.
